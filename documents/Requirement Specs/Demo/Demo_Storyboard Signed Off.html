Introduction


All of the demo actors below are fictional. Any relationship between demo actors and real ones is completely coincidental.

There are three separate demos for three different sets of people:

  • A 'referring physician' demo
  • A 'radiology department admin' demo [I'm thinking of Gordon Smith from MGH here]
  • An 'investor' demo.

Each of these demos is nested - the 'investor' one includes the 'radiology department admim' one and the 'radiology department admin' includes the 'referring physician'.

Hardware required for referring physician demo:

  • Central server
  • Router running on the internet at medcommons.net under a different port (the default 9080 works just fine)
  • Either
    • tablet/laptop
    • user's own desktop machine on the internet.
  • Working email account (MedCommons demo user's own email, nothing special)

Additional hardware required for 'radiology department admin' demo

  • Laptop with router installed.

Additional hardware required for 'investor' demo:
<none>

Note that there isn't a demo for the following people:

  • Patients
  • Radiologists

And no plans (for the demo) to show:

  • Registration
  • Installation of the router.

Demo Storyboards

Each demo has the following attributes:

  • A gestalt of what we're trying to communicate with that person.
  • A sequence of steps to show them how MedCommons software works.

Referring Physician

Gestalt: We want to show how simple it appears, how little time it takes to get started,  what information will be presented to them, and what is tracked for HIPAA puposes.

  1. Display one of three JPEG DICOM invitations (see appendix for description and a picture). This could be off of a web page or in an email in the MedCommons user's personal account.
  2. The customer clicks on the JPEG and is taken to the selection screen to allow the user to choose one study. If there is only one study for this customer then the selection screen dissolves immediately into the WADO viewer. Either way the selection screen has read a hidden file (<GUID>.secretStuff) which has all of the extra parameters for the WADO viewer and the order form.
  3. The username and other parameters will be used to (later) put an entry in the HIPAA log.
  4. The WADO/order form viewer is launched and displays the order form as the default. They see some patient history and other demographics. They are told at this time that a HIPAA log entry has been made for this visit to the MedCommons server.
  5. The user is shown how to navigate around the viewer. We show all of the features which are listed in the user manual in the 510(k) submission if the user is interested (mousewheel, zoom, pan, window/level).
  6. We show the user the HIPAA log entry by hitting the i button. It shows the latest visit as well as previous entries for this user.

The rest of the demo is verbal: describing to the referring physician that they could enter a new order form for second opinions; that the email could be sent to their patients if they thought that was clinically useful; that other services (3D, CAD) were coming to the market in the future and that they weren't going to be tied to a hospital.

What needs to be done for this to work:

  • The HIPAA log is populated with the user field.
  • The order form and viewer need to be integrated. This means:
    • When viewing the order form clicking on the image thumbnails replaces the order with the WADO viewer; clicking on the order form thumbnail replaces the viewer with the  order form.
    • Some state is saved on the server so that going back and forth between the order form and viewer is seamless. Image window/level, zoom region, and image selection remain intact.
  • Viewer bugs need to be fixed:
    • IE pan of zoom rectangle must work.
    • Tools need to work:
      • Zoom, Window/Level
      • Rotate
      • Reset
      • DICOM overlays on/off [Need to define more elements for display]
      • HIPAA log
    • Tools not needed for demo
      • Exit (to what?)
      • Comparison
      • Annotation
    • Viewer must work on a number of different screen sizes (landscape and portrait)
    • Speed should be faster.
  • Selection screen needs to detect the size of the window and send parameters down to the WADO viewer so that it can render the screen in landscape or portrait mode.
  • Invitation should be a JSP which picks the right JPEG and launches the selection screen with the right GUID parameters. It's pretty much a static page which passes the right parameters through. How much is a JPEG and what is filled in by a JSP is defined by what is easiest for the demo. The important aspect is that they be clear and attractive.

 

Radiology Department Admin

Gestalt: We want to show how simple it appears, how little time it takes to get started,  how they can create an order, how HIPAA logging works. We want them to see that this a tool which they can use for certain parts of their workflow; it also provides a simple managed way to get images from their institution to other locations or persons.

  1. Ask to plug in laptop into their ethernet. If they don't - then give them the physician demo on their desktop machine and demo as much as possible on the laptop (this means that they can see the selection screen, the WADO viewer, but they can't generate an order or look at the HIPAA log because the order form requires a GUID from Central and the HIPAA log is on Central).
  2. Show the selection screen on the router. We will have (say) 10 studies on the router that can be selected. The customer can load either an order form or the WADO viewer with a single click.
  3. If they choose the order form: fill in as many fields as possible from the DICOM demographics; other fields can be edited by hand but the defaults are in the <GUID>.secretStuff file. [Assume that the order form for SCAR is the right one to use here] the only place that the secret stuff needs to enter into the demo is at the selection screen. Everywhere downstream of there it can be carried by passing the URL parameters onto the next screen.
    The WADO viewer can be shown by clicking on an image thumbnail.
  4. If they choose the WADO viewer then move to the order form fairly quickly by clicking on the order form thumbnail. These guys need to see that the viewer exist but they aren't going to
  5. The order form has few options:
    1. CopyTo contains email addresses for people who are going to get notifications. We should encourage them to put their own email addresses there. Currently this is a no-op - it's a placeholder for sending email messages after the demo - but it's an important part of the story.
    2. Not sure of what else. There isn't any round trip or description of the type of orders that I see on the OrderForm.jsp - the one that is there is a 'Night Interpretation Order'.  We say during the demo that more types of orders will be supported in the future.
  6. When they submit the order form the user gets a confirmation screen saying that the order has been queued. Either we should show transfer progress here or have a progress element on the screen which is grayed out (to be made real after the Demo).
  7. Show that the images can be viewed off of Central (at a router running at Central).
  8. Show that launching the order form or the viewer creates HIPAA entries. There should be one HIPAA entry per session (to be defined however it is easy to do).

Right now the image transfer pretty much stops there. Encryption for transfer is mentioned but not described in detail.

Appendix A: DICOM "Invitation"

Text and picture from Adrian's email of May 7.

I think the Order Form and patent issues are finally converging  to my satisfaction as a truly viral Open Radiology method description.

The attached faux screen shot shows a random radiologist's workstation displaying the first image of the first series of a study that has been put under the control of MedCommons. As you can see, the PACS unwittingly transports DICOM study and displays the JPEG image that is the DICOM face of our encrypted Order as if it was an ultrasound or some other color image. In this case, however, the MedCommons invitation is the only series that the workstation can display because the other series are encrypted and look like noise. As you can read for yourself, the user is prompted to go to www.medcommons.net where they are given a choice of viewing the images in our WADO viewer or installing the MedCommons software so that they can view the images on their workstation or PACS.

Either way they pay nothing and we win.

Adrian

PS This scheme works even if the patient demographics are completely anonymous!
